# reso-dd-kb

A small, self-contained RESO Data Dictionary 2.0 knowledge base for
LLM agents working on the Matrix MLS / Atlas platform. Built using
Karpathy's "LLM Wiki" pattern: immutable Raw inputs, an LLM-owned
generated Wiki, and a hand-written Schema (`AGENTS.md`) that turns any
LLM into a disciplined RESO librarian.

## What's in here

| Folder | Purpose |
|---|---|
| `raw/` | Immutable upstream sources — RESO DD 2.0 spreadsheet, parsed CSVs, RESO certification stats, our description corpus, dd.reso.org scraped metadata (resource / field / lookup). |
| `wiki/resources/` | One Markdown page per RESO Resource (~41 files). Generated by `scripts/refresh.py`. |
| `wiki/dbml/reso-2.0-canonical.dbml` | Pure RESO DD 2.0 canonical DBML (41 Resources + 193 lookup ref tables + FK Refs). Generated by `scripts/build_reso_canonical_dbml.py`. |
| `wiki/atlas/atlas-target.dbml` | Atlas-adapted DBML (drops, renames, custom fields). Generated by `scripts/build_atlas_target_dbml.py`. |
| `scripts/` | Fetchers (`fetch_resource_metadata.py`, `fetch_lookup_metadata.py`, `fetch_field_metadata.py`), wiki/DBML builders (`refresh.py`, `build_reso_canonical_dbml.py`, `build_atlas_target_dbml.py`), `dump_descriptions.sh`. |
| `AGENTS.md` | Schema the LLM obeys (boundaries, query workflow, refresh workflow, citation style). |
| `methodology.md` | One-page primer: naming, keys, lookups, lifecycle, how to read `Org%`. |

## Upstream source

All metadata is scraped from [dd.reso.org/DD2.0](https://dd.reso.org/DD2.0/),
the official RESO documentation site. The legacy
[ddwiki.reso.org](https://ddwiki.reso.org/) is being retired by RESO and
this KB no longer reads it. The `WikiPageUrl` columns inside
`raw/fields.csv` and `raw/lookups.csv` (parsed from the official RESO
xlsx) still point at ddwiki and are preserved as tombstones.

> Note on `Sys%`: ddwiki used to publish a per-System adoption percentage
> in addition to per-Organization. dd.reso.org dropped the per-System
> breakdown; the KB now exposes `Org%` only.

## Why "descriptions only + dd.reso.org link-out"?

We rejected mirroring the full upstream prose. Size comparison:

| Strategy | Repo size | Refresh story | Verdict |
|---|---|---|---|
| **Descriptions only (this KB)** | ~5 MB | seconds for re-render; ~7 min for full re-scrape | **Adopted** |
| Lean MD mirror (per-field cleaned) | ~12 MB | ~10 min/refresh + monthly cron | Revisit when offline-LLM matters |
| Full MD mirror | ~30 MB | ~10 min/refresh | Diminishing returns |
| Raw HTML mirror | ~240 MB | needs Git LFS | Not recommended |

Each Resource page embeds the short canonical description plus the
structured KV (BEDES, PropertyTypes, dates). When the agent needs deep
prose, it clicks the dd.reso.org link in the Fields table.

## Refresh

```bash
# 1. drop new RESO DD xlsx into raw/
# 2. parse xlsx -> fields.csv + lookups.csv (one-off openpyxl script)
# 3. scrape dd.reso.org (incremental: If-Modified-Since)
python3 reso-dd-kb/scripts/fetch_resource_metadata.py     # ~41 pages
python3 reso-dd-kb/scripts/fetch_lookup_metadata.py       # ~290 pages
python3 reso-dd-kb/scripts/fetch_field_metadata.py        # ~1,745 pages

# 4. dump description corpus from Atlas Supabase
PSQL_URL=postgres://... bash reso-dd-kb/scripts/dump_descriptions.sh

# 5. rebuild wiki/resources/
python3 reso-dd-kb/scripts/refresh.py

# 6. rebuild canonical DBML (full RESO 2.0 schema, 41 Resources + 193 Lookups)
python3 reso-dd-kb/scripts/build_reso_canonical_dbml.py
```

All three fetchers are incremental — `If-Modified-Since` keeps
steady-state refreshes touching tens of pages instead of ~2,000.

## Using it (humans + LLMs)

- **Humans**: open `wiki/resources/<Resource>.md` and search the Fields
  table. Click the dd.reso.org link in the right-most column for full
  RESO prose. For a complete schema view, open
  `wiki/dbml/reso-2.0-canonical.dbml` in a DBML viewer (e.g.
  [dbdiagram.io](https://dbdiagram.io)).
- **LLMs**: read `AGENTS.md` first. It defines page anatomy, query
  workflow, citation style, and the strict "never invent a field"
  rule.
