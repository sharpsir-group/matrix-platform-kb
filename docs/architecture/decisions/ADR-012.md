# ADR-012: Dedicated Matrix CDL Supabase Project (separate from SSO)

**Status:** Accepted
**Date:** 2026-04-20
**Supersedes / extends:** ADR-001 (Supabase as CDL), ADR-002 (dual-Supabase)
**Related:** ADR-013 (matrix-platform-foundation owns CDL), ADR-014 (ingestion pipeline)

## Context

Until Q1 2026 the shared MLS tables (`mls_listings`, `mls_contacts`, …)
lived inside the Matrix SSO Supabase project (`xgubaguglsnokjyudgvc`).
That project also hosts authentication, `sso_users`, `sso_applications`,
`sso_roles`, and all SSO edge functions. Co-locating business data with
auth caused three problems:

1. **Blast radius.** A breaking migration on the MLS tables could take
   down SSO itself. SSO is the hardest dependency across every Matrix
   app; outages cost every team simultaneously.
2. **Lakebase portability.** We expect to migrate the CDL onto
   Databricks Lakebase (managed Postgres) within 12–18 months. Tangling
   the schema with Supabase-native auth tables (`auth.users`, SSO
   custom types) creates artificial coupling that blocks portability.
3. **Lovable risk.** Multiple Matrix apps have their own Lovable-linked
   Supabase projects and can regenerate schema at will. If any one of
   those apps were pointed at the CDL, a single Lovable prompt could
   mutate shared business data schema. The CDL must be single-writer
   at the schema level.

## Decision

Provision a **dedicated Supabase project** to be the Matrix CDL
(Common Data Layer). The project is explicitly **not linked to Lovable**.
The canonical listing tables (`public.properties`,
`public.properties_published`, `public.property_media`), the
`cdl_staging.*` raw/mapped tables, and the MLS Sync control plane
(`mls_settings`, `mls_sync_jobs`, `mls_sync_state`,
`mls_orchestrator_runs`) all live there. (See ADR-014 for the
implementation status note vs the original 18-table `mls_*` design.)

After evaluating fresh provisioning vs repurposing an existing nano
project, we repurposed `ofzcokolkeejgqfjaszq` (previously labelled
"Websites-MLS", now "Matrix Data Model Studio") as the **Matrix CDL**
project. The project was wiped (`DROP SCHEMA public CASCADE`), keys
rotated, renamed in the Supabase org UI, and bootstrapped by the
migrations in `matrix-platform-foundation/supabase/cdl/migrations/`.

### Identity boundary

- **SSO project** (`xgubaguglsnokjyudgvc`) issues all JWTs. It remains
  the sole identity authority.
- **CDL project** (`ofzcokolkeejgqfjaszq`) uses Supabase's **Third-Party
  Auth** with the SSO JWKS URL + issuer. CDL PostgREST verifies SSO
  tokens natively; CDL RLS policies read claims from `auth.jwt()` via
  the JWT helper functions ported into the CDL project's `public` schema.
- No `auth.users` row is ever created on CDL. No foreign keys point at
  `sso_users`. Every row carries `tenant_id` + optional user id string
  columns only, guaranteeing Lakebase portability.

### RLS

RLS is enabled day one on every CDL table (no "RLS later" escape
hatches). Policies follow Patterns A-E from the platform security
model, rewritten to pull context exclusively from the JWT (no
`current_setting('request.jwt.*')` server-side — that still works but
Lakebase will not honour Supabase-specific GUC hooks, so the helpers
use pure `auth.jwt()` JSON access).

### Ingestion + writes

Apps never reach the CDL service-role key. Writes flow through Edge
Functions on the CDL project (the 5-stage ingestion pipeline plus the
`mls-sync` / `mls-sync-orchestrator` admin EFs — see ADR-014 status
note and `docs/data-models/cdl-schema.md`). Reads use the CDL anon
key with RLS enforcement on `public.properties_published`, or go
through the `listings-search` EF for filtered/paginated lookups.

## Consequences

### Positive

- Hard isolation between auth and business data.
- SSO can be upgraded / restored independently of CDL.
- CDL is a clean candidate for Lakebase migration: no Supabase-native
  FKs in `public`, Supabase-specific helpers quarantined in `_infra`.
- Schema evolution is single-writer via `matrix-platform-foundation`
  (see ADR-013).

### Negative

- Two projects to monitor, secure, and rotate keys for.
- Cross-project joins (e.g. enriching a listing with the SSO user's
  display name) require a batched EF (`resolve-users`) instead of a
  SQL join. Addressed by the `useUserDisplay` React hook that
  caches batched lookups.
- All Matrix apps had to split their `dataLayerClient.ts` into
  `ssoClient` and `cdlClient`; this is a one-time migration cost.
