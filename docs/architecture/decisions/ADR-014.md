# ADR-014: Unified MLS Ingestion Pipeline (sources → staging → merge → mls_listings)

**Status:** Accepted
**Date:** 2026-04-20
**Related:** ADR-012 (dedicated CDL), ADR-004 (Databricks ETL),
ADR-005 (RESO DD 2.0), ADR-006 (RESO Web API)

## Context

The Matrix CDL must accept listing/contact data from several upstream
producers with different shapes, trust models, and cadences:

- **RESO Web API endpoints** (external MLS feeds via OData 4.01 +
  OAuth2 `client_credentials`).
- **Our own `mls_2_0` RESO Web API** (built on Databricks Gold; see
  ADR-009 medallion). The CDL treats this as just another RESO feed,
  which keeps the topology simple.
- **CSV uploads** by brokers / data_ops.
- **Generic CRM webhooks** (Qobrix, Dash, third-party CRMs) posting
  their own JSON shapes.
- **Manual entry** by brokers through the `matrix-mls` admin UI.

Previously, each integration was bespoke: separate EF, separate error
model, no shared provenance, no audit log, no deterministic dedup.
Adding a new source was a copy-paste effort prone to drift.

## Decision

Adopt a **single ingestion pipeline** with one entry model and one
merge model, applied uniformly to every source type.

```
               ┌──────────────────┐
               │ mls_sources      │  registered per-tenant per-source
               └────────┬─────────┘
                        │
                        ▼
 ┌─────────────┐   ┌──────────────────────┐
 │ reso-import │──►│ mls_staging_listings │ ◄── csv-import
 └─────────────┘   └────────┬─────────────┘ ◄── crm-import
                            │                 ◄── manual-entry (direct UI)
                            ▼
                   ┌──────────────────┐
                   │ listing-merge EF │ strategy: prefer_incoming |
                   └────────┬─────────┘            prefer_existing | reject
                            ▼
                   ┌──────────────────┐
                   │ public.mls_listings │  canonical row, provenance cols
                   └──────────────────┘

        Every decision appended to mls_import_events
        Every execution logged in mls_import_runs with cursor + counters
```

### Control-plane tables

- `mls_sources` — registry of every feed per tenant. `config` jsonb
  carries type-specific settings (RESO OAuth URL, CSV delimiter
  defaults, CRM HMAC vault key).
- `mls_source_field_mappings` — per-source mapping from source payload
  field → canonical CDL column (`canonical_field`), with an optional
  `transform` (trim, upper, lower, cast, enum_map).
- `mls_import_runs` — one row per sync execution, with `status`,
  `records_read`, `records_staged`, `records_promoted`,
  `records_updated`, `records_failed`, `cursor`, `error_summary`.
- `mls_staging_listings` — landing zone. Holds `raw_payload` and
  `mapped_payload` plus provenance (`source_id`, `source_record_id`,
  `source_modification_ts`). `validation_status` enum:
  `staged | validated | promoted | rejected | superseded`.
- `mls_import_events` — append-only audit log. Powers the Audit page.

### Edge functions (all on the CDL project, `verify_jwt: false`)

- `reso-import` — OData 4.01 pager with incremental
  `$filter=ModificationTimestamp gt <cursor>`, OAuth2
  `client_credentials` token fetched from Vault.
- `csv-import` — text CSV → staging, with row-count safety limit.
- `crm-import` — webhook-style; optional HMAC-SHA256 signature
  verified against a Vault secret keyed on the source.
- `listing-merge` — promotes staging rows into `public.mls_listings`.
  Dedup first by `(x_sm_source_id, x_sm_source_record_id)`, then by
  `(tenant_id, listing_key)`. Supports three strategies:
  - `prefer_incoming` (default for batch): update on conflict.
  - `prefer_existing`: keep existing row, mark staging rejected.
  - `reject`: same outcome as `prefer_existing` plus explicit
    operator-driven audit trail.

### Provenance contract

Every row in `public.mls_listings` / `public.mls_contacts` carries:

| column | meaning |
|---|---|
| `x_sm_source_id` | FK → `mls_sources.id` |
| `x_sm_source_record_id` | external primary key (e.g. RESO `ListingKey`) |
| `x_sm_ingested_at` | promotion timestamp |
| `x_sm_ingested_by_run_id` | FK → `mls_import_runs.id` |
| `x_sm_last_source_sync_at` | source `ModificationTimestamp` or equivalent |

A unique partial index `(x_sm_source_id, x_sm_source_record_id)`
guarantees the same external record always resolves to the same row,
which makes updates idempotent across retries.

### Authorization

Every ingestion EF accepts either:

- An SSO JWT whose roles include `data_ops`, `org_admin`, or
  `system_admin` (operator-triggered), OR
- A direct service-role bearer (for `pg_cron` schedules on CDL).

No anon ingestion.

## Consequences

### Positive

- One model for every new source. "Add a CRM" = register an
  `mls_sources` row + field mappings + point at `/crm-import?source=…`.
- Full audit trail: run, staging row, and event per decision.
- Deterministic dedup via provenance partial-unique index.
- Manual review is first-class (`validation_status = 'staged'` → admin
  can inspect before promotion).
- `mls_2_0` pipeline unchanged on its side: it continues to expose
  a RESO Web API, and CDL pulls from it via the same `reso-import`
  path that handles external MLS feeds.

### Negative

- Two-hop write (staging → merge) adds latency vs direct insert.
  Mitigated by running `listing-merge` on a short pg_cron cadence per
  source, with an immediate-merge option for manual entry.
- Every source owner must maintain field mappings. The admin UI in
  `matrix-mls` / `/admin/mls/field-mappings` exists specifically for
  this; no schema changes are required to onboard a feed.
