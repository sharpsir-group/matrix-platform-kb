# ADR-011: ES256 JWT Signing — Migration from HS256

**Status**: Accepted (in progress)
**Date**: 2026-04-09
**Supersedes**: Partial — extends ADR-002 (dual-Supabase) and ADR-008 (5-level scope)

## Context

The Sharp Matrix SSO platform originally signed all custom JWTs with **HS256 (HMAC-SHA256)** — a symmetric algorithm where the same secret is used for both signing and verification. This created several issues:

1. **Supabase PostgREST rejection**: Supabase projects that migrated to ES256 as their default signing algorithm rejected HS256 custom tokens with `PGRST301: No suitable key or wrong key type`, unless a legacy HS256 key remained in "Previously used" status.

2. **Secret distribution**: The same JWT secret had to be stored in the SSO vault AND shared with every app's Supabase project. A single compromised secret exposes all token verification.

3. **Supabase direction**: Supabase moved to ES256 (ECDSA P-256) as the default signing algorithm for new projects. HS256 is a legacy path.

4. **Key rotation**: Symmetric key rotation requires coordinated secret changes across all services simultaneously. Asymmetric keys allow public key distribution without exposing the signing key.

## Decision

**Migrate all SSO JWT signing from HS256 to ES256 (ECDSA P-256).**

### Implementation

1. Generate an ES256 key pair (P-256 curve) and store the private key as a JWK in the SSO vault (`sso_es256_signing_key`).

2. Import the ES256 public key as a **standby key** in each Supabase project that needs to verify SSO tokens. PostgREST automatically trusts standby keys for verification.

3. Update `oauth-token` and `switch-role` Edge Functions to sign with ES256 when the key is available, with HS256 as a fallback.

4. During the transition period, use a hybrid strategy: sign ES256 for apps using the SSO PostgREST directly; sign HS256 for apps with their own Supabase projects (which may not yet have the ES256 key imported).

5. Progressively import the ES256 key to all app Supabase projects, then remove the HS256 gate.

6. Promote ES256 to "current" key in all projects and retire HS256 secrets.

### Signing Decision Logic (Edge Functions)

```
if app has jwt_secret_name (own Supabase project):
    sign with HS256 using app-specific secret     ← transitional
else if ES256 vault key available:
    sign with ES256 using vault key                ← target state
else:
    sign with HS256 using SSO JWT_SECRET           ← legacy fallback
```

### Verification Order (switch-role)

```
1. ES256 (vault key)          — new asymmetric tokens
2. App HS256 (vault secret)   — app project tokens
3. SSO HS256 (JWT_SECRET env) — legacy SSO tokens
4. Opaque token lookup        — sso_access_tokens table
```

## Goal State

All SSO JWTs signed with ES256. Zero HS256 signing in production. The HS256 verification path remains for a deprecation period to accept tokens issued before the migration.

## Consequences

### Positive

- **Supabase-native**: ES256 is Supabase's default, so custom SSO tokens are indistinguishable from native tokens at the PostgREST layer.
- **No secret sharing**: Only the public key is distributed to app projects. The private key never leaves the SSO vault.
- **Standard JWKS**: Future option to expose a `/.well-known/jwks.json` endpoint for third-party integrations.
- **Key rotation**: Can rotate by generating a new key pair and importing the public key as standby, with zero downtime.
- **Lovable compatible**: Lovable apps use the Supabase JS client which is algorithm-agnostic — it just passes the token string.

### Negative

- **Transition complexity**: During hybrid phase, some tokens are ES256 and some HS256 depending on the app. Edge Functions must handle both.
- **Key import**: Each app's Supabase project requires a manual key import via the Dashboard (Management API support for standby keys is limited).
- **Vault dependency**: ES256 signing requires a successful vault read on every token issuance. If the vault is unavailable, signing falls back to HS256.

### Neutral

- **RLS helpers are algorithm-agnostic**: PostgreSQL RLS functions read `current_setting('request.jwt.claims')` which contains the decoded JWT payload regardless of signing algorithm. No RLS changes needed.
- **No app code changes**: Frontend apps store and pass the token string. The signing algorithm is transparent to application code.
