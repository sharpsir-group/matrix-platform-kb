# ADR-013: `matrix-platform-foundation` Owns Both SSO and CDL Supabase Projects

**Status:** Accepted
**Date:** 2026-04-20
**Related:** ADR-012 (dedicated CDL project), ADR-003 (Lovable as app builder)

## Context

Matrix has a fleet of Lovable-linked apps that each own their own
per-app Supabase project. Lovable regenerates schema frequently based
on product prompts, which is fine for an app-specific database but
catastrophic for a shared database that dozens of apps read and write.

> **Note (2026-04):** `matrix-mls` (app DB `wckwfbbqiupvallmhqbu`) was
> originally Lovable-linked like its peers but has since been detached
> from Lovable and is now Cursor-managed. The boundary rules below
> still apply â€” they are just enforced via git/PR review rather than
> by Lovable's sandbox.

Two projects in our stack are **shared** and must never be Lovable-owned:

1. The **SSO** Supabase project (`xgubaguglsnokjyudgvc`) â€” identity
   authority for every Matrix app.
2. The **CDL** Supabase project (`ofzcokolkeejgqfjaszq`, per ADR-012)
   â€” single source of truth for MLS / listing / contact / transaction
   data consumed by many apps.

We also tried having a dedicated Lovable app (`matrix-cdl-studio`) act
as the schema owner for the CDL. That app's Lovable regenerations were
an unpredictable source of schema drift â€” exactly the failure mode we
want to eliminate.

## Decision

**Both the SSO and CDL Supabase projects are owned and managed
exclusively by the `matrix-platform-foundation` repository.**

- All migrations and Edge Functions for SSO live in
  `matrix-platform-foundation/supabase/`.
- All migrations and Edge Functions for CDL live in
  `matrix-platform-foundation/supabase-cdl/` (a sibling folder so that
  one repo can push to two projects; `config.toml` in each subfolder
  pins the respective `project_id`).
- Lovable is **not linked** to either project. Schema changes are
  proposed as PRs against `matrix-platform-foundation`, reviewed by
  platform engineers, and pushed via `supabase db push --project-ref ...`
  in CI or by the on-call platform engineer.
- `matrix-cdl-studio` is retired (see retirement notice in that repo).
  Any future "CDL inspector" app must be read-only with the CDL anon
  key; it cannot and will not own schema.

### Why not two repos?

- Both projects share the same ADRs, JWT helpers (ADR-011 ES256 keys),
  and the same operators. Splitting would duplicate the ADR/runbook
  workflow and increase the risk of drift between the JWT helper
  implementations.
- SSO â†” CDL changes are frequently coordinated (e.g. adding a new
  admin page key to `sso_app_page_catalog` on SSO while adding the
  new `mls_*` admin table on CDL). Single-repo PRs keep those changes
  atomic and auditable.

### Boundary rules

- An app's own per-app Supabase project stays under that app's repo
  (typically Lovable-linked; `matrix-mls` @ `wckwfbbqiupvallmhqbu` is
  the exception â€” Cursor-managed after the cutover). It holds
  app-specific tables only (`role_configurations`, app-local audit
  logs, etc.).
- Apps **never** reach the CDL service-role key. CDL writes go through
  `cdl-write` (CORS proxy) and the ingestion EFs (ADR-014).
- Apps never reach the SSO service-role key either; SSO admin actions
  happen through the SSO admin EFs.

## Consequences

### Positive

- Single auditable repo for the two highest-blast-radius Supabase
  projects in the platform.
- Easy to enforce: if a Lovable prompt tries to change the CDL, the
  project isn't linked and the prompt can't.
- Clean story for ADRs and runbooks â€” both shared projects live next
  to each other in one tree.

### Negative

- `matrix-platform-foundation` contributors need access to two
  Supabase projects (we treat this as a platform-team privilege).
- Contributors writing admin UI in `matrix-mls` must remember the
  split: app migrations go in `matrix-mls/supabase/`, but the shared
  SSO role / page catalog changes go in
  `matrix-platform-foundation/supabase/`.
