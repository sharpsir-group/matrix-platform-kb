# ADR-002: Why dual-Supabase (SSO + per-app DB) over schema-based multi-tenancy

## Status

Accepted — **partially superseded by ADR-012 / ADR-013 (Apr 2026)**.

> The "CDL lives inside the SSO project" portion of this ADR is obsolete.
> The Matrix CDL is now its own dedicated Supabase project
> (`ofzcokolkeejgqfjaszq`) and CDL authentication uses Supabase
> **Third-Party Auth** against the SSO JWKS — no "Supabase native
> token" path, no `app_metadata` fallback on CDL helpers. See
> [ADR-012](ADR-012.md) and [ADR-013](ADR-013.md) for the current
> topology. The "one SSO + one per-app DB" pattern and the token flows
> described below for **per-app DBs** remain correct.

## Context

The platform serves 12 applications that require shared authentication but separate business data. Three approaches were considered:

- **Single DB with schemas**: One PostgreSQL instance, separate schemas per app
- **Single DB with RLS**: One instance, RLS policies filter by app/tenant
- **Separate DBs**: One SSO instance + one app-specific instance per application

Concerns included: identity isolation, data sovereignty, and the ability to scale or migrate individual apps independently.

## Decision

Use **two Supabase instances per app**:

1. **SSO instance** (shared): Auth, users, groups, permissions, app registry
2. **App DB instance** (per app): Business data (listings, contacts, transactions, etc.)

Each app configures two Supabase clients:

- **App DB client** — injects the SSO JWT via `accessToken` hook. PostgREST validates the token and RLS policies read custom claims (`scope`, `crud`, `uoi`, `team_ids`) from `current_setting('request.jwt.claims')`.
- **CDL client** (CDL-Connected apps only) — uses the **Supabase native token** (signed by the CDL project's own key) for PostgREST calls. Since native tokens lack custom SSO claims, CDL RLS helpers fall back to `auth.users.raw_app_meta_data` where `oauth-token` persists `active_scope`, `active_crud`, and `active_team_ids` at login/refresh.

### Token Architecture (Two-Token Model)

| Token | Signed With | Used For | Custom Claims |
|-------|------------|----------|---------------|
| **SSO JWT** | ES256 (SSO key) or HS256 (app secret) | App DB PostgREST, Edge Functions | Full: scope, crud, uoi, teams, sso_role |
| **Supabase native token** | Project's own key (ES256) | CDL PostgREST | None — falls back to `app_metadata` |

> **Critical**: CDL RLS helper functions (`get_active_scope()`, `get_crud()`, `get_current_team_ids()`) **must** include `app_metadata` fallback. Without it, native tokens return NULL for all custom claims and RLS blocks all access. See [security-model.md](../../platform/security-model.md#app_metadata-fallback-cdl-instance).

> **Goal**: Once ES256 keys are imported into all project instances, SSO JWTs will be accepted by all PostgREST instances natively, eliminating the need for the two-token model and `app_metadata` fallback. See [ADR-011](ADR-011.md).

## Consequences

**Positive**

- Clean separation of identity (SSO) and domain data (App DB)
- Apps can be deployed to different regions or vendors without touching auth
- SSO instance can be scaled/backed up independently
- Simpler RLS: App DB policies only handle business scope, not auth plumbing

**Negative**

- Extra client configuration per app — two URLs, two keys
- JWT must be valid and trusted by both instances; custom claims must be consistent
- CDL-Connected apps require `app_metadata` persistence and fallback in RLS helpers (added complexity)
- Cross-instance queries (e.g., user + listing) require two round-trips or an aggregation layer
