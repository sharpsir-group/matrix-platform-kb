# Personalization & Recommendation Engine

> **For Lovable**: This spec defines how personalized property recommendations work.
> Phase 4 feature (BI + AI Launch, Jul-Oct 2026). Spec it now to guide data collection decisions.

---

## Visitor Profiling (Anonymous, Pre-Auth)

| Category | Attributes |
|----------|------------|
| **System** | IP geolocation (country, city), device type, OS/browser, datetime/timezone, referral source |
| **Behavioral** | Pages visited, listings viewed, search queries, time-on-page, scroll depth, click patterns |

**Storage**: Supabase `visitor_sessions` table with anonymous session ID (cookie), linked to user on login. Session merges on login: anonymous events attach to user record for preference building.

**Privacy**: GDPR consent required before tracking (cookie banner). Retention: 90 days anonymous; linked to user for authenticated. See [compliance.md](../platform/compliance.md) for DSAR and retention rules. No behavioral tracking before consent.

## Semantic Ranking (Listing Personalization)

| Component | Implementation |
|-----------|-----------------|
| **Embeddings** | `properties_published.embedding vector(1024)` (pgvector). Generated by `embed-properties` cron from title + description + city + property_type. See [intelligence-layer.md](../architecture/intelligence-layer.md). |
| **Structured feature vector** | `properties_published.feature_vector vector(64)` for normalized numeric features (price band, bedrooms, area, recency) |
| **User preference vector** | Computed from `internet_tracking_events` (RESO Actor / Event / Object groups) — viewed listings, favorites, search queries — weighted by event type |
| **Ranking formula** | `score = α × semantic_similarity + β × recency + γ × price_fit + δ × location_proximity` |
| **Cold-start fallback** | Popularity-based ranking by market, then property type |

Alpha/beta/gamma/delta weights tuned per market (Cyprus, Hungary, Kazakhstan).

## Recommendation Engine

| Type | Method | Use Case |
|------|--------|----------|
| Similar listings | Embedding cosine similarity (`embedding <=> embedding`) | "More like this" on listing detail |
| You might also like | Collaborative filtering on `internet_tracking_events` (actor × object × event) | Cross-user preference patterns |
| Best agent for this listing | Cross-resource cosine vs `members.embedding` | Listing-detail "Talk to specialist" |
| New matches | Saved search criteria | Push notifications when new listings match |

**Integration points**: Client Portal (auth), Website (anon + auth), Broker App (client recs), MCP server (agent copilots). See [ecosystem-architecture.md](../platform/ecosystem-architecture.md) for channel diagram.

**Cold-start**: New visitors get popularity-based ranking. After 3+ listing views or 1 saved search, switch to semantic + collaborative signals.

## Data Collection Architecture

```
Client-side event tracker → Edge Function `track-visitor-event` → public.internet_tracking_events
```

The CDL stores all behavioral data in **RESO-canonical** tables (no invented schemas), so the same data feeds recsys, BI, and outbound channels:

| CDL Table | Purpose |
|-----------|---------|
| `internet_tracking_events` | RESO Actor / Event / Object groups: anonymous + authenticated session, system attributes, behavioral events (view, search, click, favorite, inquiry, share). Cookie-stable `actor_mls_id` for anonymous; `actor_key` after login. |
| `properties_published.embedding vector(1024)` | Listing description embedding for semantic search |
| `properties_published.feature_vector vector(64)` | Structured numeric features for ranking |
| `properties_published.marketing_metadata jsonb` | Per-channel campaign data (UTM defaults, social-share overrides) |
| `members.embedding vector(1024)` | Agent bio embedding for "best agent" matching |
| `recommendations(actor_key, listing_keys text[], computed_at)` | Nightly recsys output read by surfaces |

**Sync**: Embedding cron (`embed-properties`) refreshes vectors nightly or on listing publish. Recsys cron (`recommendations-cron`) runs ALS/implicit-MF nightly over the last 90 days of `internet_tracking_events` and writes `recommendations`. CDL → Databricks for offline model training and analytics.

> **Reconciliation note (Apr 2026)**: An earlier draft of this spec proposed app-local tables `visitor_sessions`, `visitor_events`, `user_preferences`, `listing_embeddings`. The implemented CDL uses **RESO `InternetTracking`** instead of `visitor_*`/`user_preferences`, and stores embeddings **inline on `properties_published`** rather than in a separate `listing_embeddings` table. This is non-negotiable for syndication and partner-API alignment.

## Implementation Phasing

| Phase | Scope |
|-------|-------|
| P1 (Phase-1 of CDL plan, Apr 2026) | `internet_tracking_events` table (RESO), `properties_published.embedding` + `feature_vector` + `marketing_metadata` columns, `members.embedding` column, cookie consent, `track-visitor-event` Edge Function. **Capture data; defer modeling.** |
| P2 (Phase-2 intelligence layer) | `embed-properties` cron, HNSW indexes, `semantic-search` EF, "Similar listings" block, MCP server, JSON-LD / IDX / Zillow syndication views |
| P3 (BI + AI Launch, Q3 2026) | `recommendations-cron` (ALS / implicit-MF), "You might also like", "Best agent for this listing", saved-search push notifications |
